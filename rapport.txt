1.1 Contexte du projet
Avec la généralisation des infrastructures cloud et des environnements hybrides, la supervision des systèmes informatiques est devenue un enjeu majeur pour garantir la disponibilité, la performance et la sécurité des services. Dans un contexte professionnel, il est indispensable de disposer d’une solution centralisée capable de surveiller plusieurs types de systèmes d’exploitation et d’anticiper les incidents.
Amazon Web Services (AWS) fournit une plateforme cloud robuste permettant de déployer des infrastructures flexibles et évolutives. Cependant, la supervision native ne couvre pas toujours l’ensemble des besoins opérationnels, ce qui rend nécessaire l’utilisation d’outils spécialisés.
________________________________________
1.2 Objectifs du projet
L’objectif principal de ce projet est de mettre en œuvre une infrastructure de supervision centralisée sous AWS en utilisant Zabbix, déployé sous forme de conteneurs Docker.
Les objectifs spécifiques sont :
•	Déployer un serveur de supervision Zabbix sur AWS
•	Superviser un parc hybride composé de machines Linux et Windows
•	Centraliser la collecte des métriques système (CPU, RAM, disponibilité)
•	Visualiser les données en temps réel à travers des tableaux de bord
•	Mettre en place une architecture simple, sécurisée et conforme aux limitations du Learner Lab
________________________________________
1.3 Outils et technologies utilisés
•	AWS : EC2, VPC, Security Groups
•	Docker & Docker Compose : Conteneurisation de Zabbix
•	Zabbix : Outil de supervision open source
•	Ubuntu Server : Serveur Zabbix et client Linux
•	Windows Server : Client Windows

2. ARCHITECTURE RÉSEAU AWS

2.1 Architecture générale et logique du projet
 
Cette architecture illustre la mise en place d'une solution de monitoring centralisée sur AWS utilisant Zabbix déployé via Docker.
•	Infrastructure Réseau : L'ensemble des ressources est déployé au sein d'un VPC unique avec un sous-réseau public pour faciliter l'accès. Une Internet Gateway (IGW) assure la connectivité vers l'extérieur.
•	Sécurité : Un Security Group centralise le contrôle des flux, autorisant l'accès Web (80/443), l'administration (SSH/RDP) et la communication entre le serveur et les agents Zabbix (10050/10051).
•	Composants du Parc :
o	Serveur Zabbix (t3.large) : Héberge le moteur de supervision, l'interface Web et la base de données sous forme de conteneurs Docker.
o	Clients Monitorés : Un parc hybride composé d'une instance Linux (t3.medium) et d'une instance Windows Server (t3.large), chacune équipée d'un agent Zabbix pour la collecte des métriques.
2.2 Présentation de l’architecture réseau
L’architecture réseau mise en place repose sur un Virtual Private Cloud (VPC) unique afin de centraliser les ressources. Un sous-réseau public est utilisé pour simplifier l’accès aux instances sans configuration de VPN.
Une Internet Gateway est associée au VPC afin de permettre l’accès Internet aux instances EC2.
 Figure 1 : Création du VPC
 
Dans le cadre de ce projet, un Virtual Private Cloud (VPC) dédié a été créé afin d’isoler et de sécuriser l’ensemble des ressources AWS utilisées pour la solution de supervision Zabbix.
Le VPC, nommé « Youssouf-VPC-Zabbix », utilise un bloc d’adressage IPv4 privé afin de permettre la communication interne entre les différentes instances EC2. Ce choix permet une meilleure maîtrise du réseau, une segmentation claire des ressources et une conformité aux bonnes pratiques AWS.
Figure 2 : Configuration du Subnet public
 
Un sous-réseau public a été créé à l’intérieur du VPC afin d’héberger les instances EC2 du projet. Ce subnet permet aux machines d’obtenir une adresse IP privée tout en étant accessibles depuis Internet grâce à une passerelle Internet.
Le subnet public est associé à une zone de disponibilité spécifique afin d’assurer une meilleure organisation des ressources et une cohérence de l’architecture réseau.
 
Une Internet Gateway a été créée et associée au VPC afin de permettre la communication entre les ressources internes et Internet. Cette passerelle est indispensable pour garantir l’accessibilité des instances EC2 déployées dans le sous-réseau public.
 
Une table de routage publique, nommée « Youssouf-RT-Public », a été créée afin de permettre aux instances EC2 du subnet public d’accéder à Internet. Une route par défaut redirige tout le trafic externe vers l’Internet Gateway associée au VPC.
Le subnet public a ensuite été explicitement associé à cette table de routage, garantissant ainsi la connectivité Internet nécessaire pour les serveurs Zabbix et les clients monitorés.
 
________________________________________
2.3 Sécurité et contrôle des accès
La sécurité de l’infrastructure est assurée par des Security Groups configurés selon le principe du moindre privilège. Les ports ouverts sont strictement limités aux besoins du projet.
Port	Protocole	Description
22	TCP	Accès SSH (Linux)
3389	TCP	Accès RDP (Windows)
80 / 443	TCP	Interface Web Zabbix
10050	TCP	Agent Zabbix
10051	TCP	Serveur Zabbix
 Figure 3 : Configuration du Security Group
 
Un Security Group nommé « Youssouf-SG-Zabbix » a été créé pour contrôler l’accès au serveur Zabbix et aux clients monitorés. Les ports nécessaires pour l’administration (SSH/RDP), l’accès Web (HTTP/HTTPS) et la communication Zabbix (10050/10051) ont été configurés selon le principe du moindre privilège. Les règles sortantes autorisent tout le trafic, ce qui est suffisant pour le fonctionnement normal du monitoring dans le lab.

5. CONFIGURATION DES CLIENTS (AGENTS ZABBIX)
5.1 Installation de l’agent Zabbix sur Linux
Sur le client Linux, l’agent Zabbix est installé et configuré pour communiquer avec le serveur Zabbix. L’adresse IP du serveur est définie dans le fichier de configuration.
Voici les étapes de la configuration
1.	Se connecter au client Linux via SSH :
ssh -i key-zabbix-lab.pem ubuntu@<IP_CLIENT_LINUX>
2.	Installer l’agent Zabbix :
sudo apt update
sudo apt install zabbix-agent -y
3.	Configurer l’agent pour se connecter au serveur Zabbix :
•	Ouvrir le fichier de configuration :
sudo nano /etc/zabbix/zabbix_agentd.conf
•	Modifier les lignes principales :
Server=<IP_PUBLIQUE_DU_SERVEUR_ZABBIX>
ServerActive=<IP_PUBLIQUE_DU_SERVEUR_ZABBIX>
Hostname=Client-Linux
•	Sauvegarder et quitter (Ctrl+O puis Ctrl+X).
4.	Redémarrer et activer l’agent :
sudo systemctl restart zabbix-agent
sudo systemctl enable zabbix-agent
5.	Vérifier que l’agent fonctionne :
sudo systemctl status zabbix-agent

6. SUPERVISION ET VISUALISATION DES DONNÉES
Cette section présente la mise en œuvre effective de la supervision à l’aide de Zabbix. Elle permet de vérifier que les agents installés sur les machines clientes Linux et Windows communiquent correctement avec le serveur Zabbix et que les métriques système sont collectées et visualisées en temps réel.
________________________________________
6.1 Ajout des hôtes dans Zabbix
Après le déploiement du serveur Zabbix et l’installation des agents sur les machines clientes, chaque hôte est déclaré dans l’interface d’administration afin d’activer la supervision.
6.1.1 Ajout du client Linux
Le client Linux est ajouté depuis l’interface Web de Zabbix via le menu Data Collection → Hosts.
L’hôte est associé au groupe Linux servers et configuré avec une interface de type Zabbix agent pointant vers l’adresse IP de la machine cliente.
Le template Template OS Linux by Zabbix agent est ensuite appliqué afin de charger automatiquement les métriques standards.
 
Après quelques instants, l’agent apparaît avec le statut Available, confirmant une communication réussie entre le serveur Zabbix et le client Linux.
Figure 10 : Client Linux ajouté et disponible dans Zabbix
 
________________________________________
6.1.2 Ajout du client Windows
De manière similaire, le client Windows est ajouté dans Zabbix et associé au groupe Windows servers.
Le template Template OS Windows by Zabbix agent est appliqué afin de superviser les ressources système spécifiques à l’environnement Windows.
 
Le statut Available de l’agent confirme que le serveur Zabbix reçoit correctement les données depuis la machine Windows.
Figure 11 : Client Windows ajouté et disponible dans Zabbix
 
________________________________________
6.2 Collecte et affichage des métriques
Une fois les hôtes configurés, Zabbix commence automatiquement la collecte des métriques système à partir des agents installés.
Les principales données supervisées incluent :
•	L’utilisation du processeur (CPU)
•	L’utilisation de la mémoire (RAM)
•	L’espace disque disponible
•	La charge système
•	La disponibilité réseau
Ces informations sont consultables via le menu Monitoring → Hosts → Latest data, permettant une visualisation détaillée et en temps réel de l’état des machines supervisées.

Figure 13 : Données récentes du client Windows

6.3 Tableaux de bord de supervision
Zabbix propose des tableaux de bord dynamiques permettant de visualiser les métriques sous forme graphique.
Un tableau de bord est utilisé pour afficher :
•	L’état de disponibilité des hôtes
•	L’utilisation du CPU et de la mémoire
•	L’occupation des disques
Ces tableaux de bord facilitent l’analyse globale de l’infrastructure supervisée et permettent une détection rapide des anomalies.
Figure 14 : Tableau de bord Zabbix affichant les métriques des clients Linux et Windows (graphique de charge CPU des Clients)

________________________________________
6.4 Détection des anomalies et alertes
Zabbix intègre un système de déclencheurs (triggers) permettant de détecter automatiquement des situations anormales, telles que :
•	Une surcharge du processeur
•	Un manque d’espace disque
•	L’indisponibilité d’un agent
Les alertes générées sont consultables dans le menu Monitoring → Problems, ce qui confirme la capacité de la solution à assurer une supervision proactive de l’infrastructure.

7. CONCLUSION
Ce projet a permis de concevoir et de déployer une infrastructure de supervision centralisée sur AWS, capable de gérer un environnement hybride composé de systèmes Linux et Windows. Le choix de Zabbix, déployé via Docker, a simplifié l’installation, la maintenance et l’évolution de la solution, tout en illustrant les bonnes pratiques de conteneurisation et de supervision professionnelle.
Plusieurs difficultés techniques ont été rencontrées au cours du projet. Le serveur Zabbix ne parvenait initialement pas à se connecter à son propre agent, des erreurs de permissions MySQL sont apparues lors de l’installation avec Docker, la visualisation des métriques sur Windows n’était pas immédiatement intuitive, et la gestion des templates ainsi que leur assignation correcte aux hôtes nécessitaient des ajustements précis. Ces problèmes ont été résolus grâce à diverses interventions, telles que la suppression de l’enregistrement du serveur lui-même, la reconfiguration de Docker et de MySQL avec les images appropriées, la vérification de l’ouverture du port 10050 sur chaque hôte, ainsi que l’attribution correcte des templates Linux et Windows afin d’assurer une collecte complète des métriques. L’utilisation des dashboards Zabbix a également permis de centraliser l’affichage des métriques et des alertes en temps réel.
Le bilan final de ce projet est très positif. La solution Zabbix fonctionne correctement et assure une supervision complète des clients Linux et Windows, permettant la collecte et la visualisation en temps réel des métriques clés telles que l’utilisation du processeur, de la mémoire, de l’espace disque et de la charge système. Le système d’alertes automatiques facilite la détection rapide des anomalies et permet d’intervenir de manière proactive. Ce projet a également contribué au renforcement des compétences en cloud computing, en supervision des systèmes, en gestion de conteneurs et en détection d’anomalies, tout en illustrant la mise en œuvre concrète des concepts théoriques étudiés.
En somme, l’infrastructure de supervision déployée s’est révélée robuste, scalable et adaptée à un usage professionnel, offrant une visibilité complète et proactive sur une infrastructure réseau mixte et permettant une gestion efficace des performances et de la sécurité du système.
